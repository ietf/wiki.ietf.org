BGP-LS Documentation Adoption Criteria

BGP-LS (RFC9552) supports a wide variety of use cases.  Work on BGP-LS typically involves adding either NLRI types or BGP-LS Attribute properties for the following use cases:

General-purpose:

  • Core BGP-LS functionality covering the protocol, route distribution, or similar common procedures.
  • BGP “generic” functionality that is broadly applicable to a number of use cases.

Application-specific:

  • Mapping link-state protocol mechanisms into a form that can be distributed by BGP-LS.  This typically covers work standardized in the LSR working group.
  • Supporting Segment Routing use cases standardized in the SPRING working group.
  • Support for the Link-State Vector Routing use cases. This work is typically done in the LSVR working group.

Work covering general-purpose use cases for BGP-LS will be done in the IDR working group and is expected to require no specific coordination with other working groups.

Adoption Criteria

For application-specific cases, there are additional requirements:

  • LSVR extensions are standardized in the LSVR working group, and may be additionally reviewed by IDR when LSVR use cases impact more general-purpose BGP-LS activities.
  • For link-state related features typically standardized in LSR, IDR will adopt work that has either an RFC, or adopted internet-drafts in the LSR working group.  Such work will be reviewed with LSR but will progress in IDR.  Last call for such documents will be done in IDR  notifying LSR.
  • For segment routing related features, SPRING does not do routing protocol work.  For features and use cases typically standardized in SPRING, IDR will adopt work where the use cases in SPRING have advanced sufficiently to justify protocol encodings.  Such work will be reviewed with SPRING.  Last call for such documents will be done in IDR notifying SPRING.

IETF Meeting Presentation Criteria

General-purpose BGP-LS features and use cases are always in-scope for IETF presentations.

Application-specific BGP-LS features and use cases should have progressed through initial discussion stages in their relevant technology working groups prior to being presented in IDR.  The motivation for this criteria is the relevant expertise for reviewing such use cases does not necessarily exist in IDR and is not followed by IDR's general membership.  Once the use cases are appropriately understood and partially specified for that technology, the expertise of IDR's membership can be used to help specify and review the BGP-LS encodings for that technology.

Presentations for new application-specific proposals that haven't had review by their technology working group are strongly discouraged from being given in IDR during the same IETF when the use case is presented for the application-specific technology the first time.  Strong cross-working group mailing list discussion is a good reason to relax this restriction.
